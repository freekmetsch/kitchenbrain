Kitchen Timeline

A cooking-first recipe page with a compact counter tray, numbered steps, semantic streams, live swaps, and a safe source-to-working-recipe migration.

Plan ready · R3 · awaiting /run

The product contract

Cooking is the default.
Source and maintenance remain close, but never precede the main kitchen task.
Position, not completion.
Selecting a numbered card moves current position and centers it. Nothing is checked off or collapsed.
Counter means untouched.
Ingredients can be checked as gathered. Chopping and measuring stay in numbered steps.
Streams can merge.
Base, filling, and topping keep stable rails; their merge creates a distinct result color and label.

One recipe truth, two supporting records

The existing recipe row remains the editable working recipe. The migration adds an immutable source snapshot for provenance and stable direction IDs for safe semantic links. cook_mode_json v5 remains validated structure—not a second recipe.

The app owns

  • UI blocks and responsive layout
  • colors and merge rendering
  • checks, swaps, current position
  • servings, timers, persistence
  • validation, fallback, accessibility

AI may author

  • clearer working directions
  • semantic streams and merges
  • ingredient-to-step allocations
  • missing cooking actions
  • timer purpose and context
R3 boundary: canonical AI promotion increments contentRevision; cache-only structure writes validate the revision but do not increment it. Both are compare-and-swap writes.
Always usable: pending, failed, invalid, or stale AI output leaves deterministic source-derived numbered cards on screen.

Release gate

Before application code consumes the new model, `/run` rehearses the append-only migration on a copied legacy database and compares recipe counts, IDs, foreign keys, and canonical ingredient/direction hashes.

Required: npm run check, npm run test:unit, npm run build, then cost-free browser stories at 375, 768, and 1280 px in Dutch and English.

Rollback keeps the additive columns, stops v5 scheduling, and returns to the deterministic/v4 projection. There is no destructive down migration.
